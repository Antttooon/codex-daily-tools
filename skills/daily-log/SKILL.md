# Daily Log

Проанализируй текущий Codex-тред и запиши только фактически выполненную работу.

## Язык

Весь отчёт и все ответы пользователю пиши **только на русском языке**.

Не переводи:

- названия проектов;
- названия технологий;
- имена файлов;
- функции;
- классы;
- переменные;
- команды.

## Что включать

Включай только:

- реализованные задачи;
- выполненные изменения;
- исправленные ошибки;
- законченные результаты.

Не включай:

- обсуждения;
- рассуждения;
- планы;
- промежуточные попытки;
- невыполненные задачи;
- повторяющуюся информацию.

## Определение проекта

Определи проект в следующем порядке:

1. Название Git-репозитория.
2. Название текущей рабочей директории.
3. Название проекта, явно указанное в текущем треде.

## Название задачи

Используй следующий приоритет:

1. Название текущего чата.
2. Если название недоступно — придумай короткое название длиной 3–7 слов.

При повторном запуске для того же треда не изменяй уже существующее название задачи.

## Дата

По умолчанию используй текущую локальную дату.

Если пользователь явно указал дату (например "за вчера" или `--date YYYY-MM-DD`), используй её.

## Файл

Храни отчёты в:

`~/codex-daily/YYYY-MM-DD.md`

Если файл отсутствует — создай его.

## Формат

Используй только такой формат:

```markdown
# YYYY-MM-DD

> Итого по задачам: <duration> · учтено задач: <count>

## <Название проекта>

### <Название задачи>

- Добавлено ...
- Исправлено ...
- Реализовано ...
- Обновлено ...

<!--
thread: <thread-id>
task: <stable-task-id>
parent: <parent-project>
started: <ISO-8601 datetime>
finished: <ISO-8601 datetime>
duration: <human-readable duration>
updated: <ISO-8601 datetime>
-->
```

## Правила записи

Перед записью:

1. Прочитай существующий файл.
2. Найди раздел проекта.
3. Если раздел отсутствует — создай его.
4. Найди комментарий с текущим `thread`.
5. После изменений пересчитай строку `> Итого по задачам: ...` под заголовком даты.

Один `thread` может содержать несколько разных задач.

Разделяй задачи, если:

- работа относится к разным проектам;
- изменился основной результат работы;
- старый блок уже стал смесью несвязанных итогов.

Для каждой отдельной задачи используй отдельный блок `###` и отдельный служебный комментарий с тем же `thread`, но разным `task`.

`task` — короткий стабильный id на латинице, например:

- `fix-auth-redirect`
- `obsidian-daily-export`
- `daily-log-style`

## Время выполнения

Для каждой отдельной задачи считай время отдельно.

Заполняй поля:

- `started` — когда работа над этой задачей началась в текущем треде;
- `finished` — когда фактическая работа над этой задачей завершилась;
- `duration` — разница между `started` и `finished`.

Используй timestamp сообщений и действий текущего Codex-треда.

Если точное начало задачи отделить невозможно, используй первый timestamp треда, относящийся к этой задаче.

Если задача уже существовала, не меняй `started`, кроме случаев когда найдено более точное или более раннее начало.

При обновлении задачи обновляй `finished`, `duration` и `updated`.

Суммарное время считай по всем задачам файла, у которых есть поле `duration`.
Если часть задач без `duration`, не включай их в сумму и в `count`.

Пиши `duration` кратко:

- `15м`
- `1ч 20м`
- `1д 2ч 5м`

Определяй связь проектов с помощью LLM-контекста:

- если задача относится к сервису, подпроекту или части большого проекта, укажи `parent`;
- пример: `web-app`, `api-service`, `worker-service` относятся к `product-suite`;
- пример: `codex-tools` не относится к `product-suite`, если работа была про локальные инструменты Codex/Obsidian;
- если parent неочевиден, не указывай его;
- не меняй название проекта на parent: сохраняй конкретный проект в `##`, а parent пиши только в комментарии.

Если задача уже существует:

- найди её по паре `thread` + `task`;
- обнови только её;
- добавь новые выполненные пункты;
- не дублируй существующие;
- обнови поля `finished`, `duration` и `updated`;
- не изменяй название задачи;
- не изменяй другие задачи и проекты.

Если задача отсутствует:

- создай новую задачу внутри соответствующего проекта;
- добавь служебный комментарий.

Если существующий блок с этим `thread` объединяет несколько разных задач, раздели его на несколько блоков с разными `task`, сохрани фактически выполненные пункты и не дублируй их.

Никогда:

- не перезаписывай весь файл;
- не удаляй существующие записи;
- не изменяй другие проекты;
- не объединяй разные задачи в одну.

## Стиль

Пиши максимально кратко.

Пиши человекочитаемо:

- описывай пользу или итог, а не внутреннюю механику;
- не используй редкие технические слова без необходимости;
- оставляй имена файлов, endpoint, commit и команды только если они помогают быстро найти изменение;
- избегай формулировок вроде "добавлен allowlist matcher", если можно написать "добавлено разрешение для локального адреса";
- название задачи делай понятным человеку, а не похожим на имя бага или файла.

Каждый пункт должен содержать один законченный результат.

Начинай пункты с глаголов:

- Добавлено
- Исправлено
- Реализовано
- Обновлено
- Настроено
- Оптимизировано
- Переработано
- Удалено

Избегай длинных предложений.

Не используй вводный текст.

## Ответ пользователю

После успешной записи ответь одной строкой:

`Отчёт обновлён: <путь>`

или

`Отчёт создан: <путь>`
